# soudoku-novel-builder 改修設計

作成日: 2026-03-10
対象: `soudoku-novel-builder` スキルと、それを支える `scripts/` およびモバイル Web ビルド
入力原稿: `草稿/初稿.md`

## 1. 依頼の理解

今回の変更は、現行スキルの前提を次のように切り替えるものと理解している。

- 変更前:
  企画から始め、プロット、キャラ、背景、章構成、本文、節画像、Web ビルドを段階的に作る。
- 変更後:
  すでに完成している原稿を起点に、構造化、抽出、画像生成、モバイル向け閲覧体験の構築を行う。

今回やりたいことは、具体的には次の一連の流れである。

1. `草稿/初稿.md` を読み、章・話の構造を抽出する。
2. 原稿からキャラクター設定を抽出して整理する。
3. 各章のあらすじを作る。
4. 原稿本文を Web ビルドしやすい Markdown に整形する。
5. キャラクターのイメージ画像を生成する。
6. 章ごとの扉絵を生成する。
7. 節ベースではなく `章 > 話` ベースのモバイル Web UI にビルドする。
8. 本文表示時は、文章を読みやすい背景 SVG / 背景表現を使う。

## 2. 原稿を見て確認したこと

`草稿/初稿.md` はすでに長編として成立しており、少なくとも以下の構造を持っている。

- 章見出し: `第一章` から `第二十一章`
- 話見出し: `第一話` から `第百十二話`
- 明示的な `節` 見出しは存在しない

つまり、今回の入力原稿は現行スキルの `章 > 話 > 節` 前提とは噛み合っていない。現状のままでは、既存の「節画像中心」の設計と Web ビルドが過剰であり、要件にも合っていない。

## 3. 変更の中核

この改修の本質は、スキルの主目的を次のように置き換えること。

- 旧: 小説を計画しながら書き進める制作スキル
- 新: 完成済み原稿を出版可能な読書体験へ変換する制作スキル

したがって、スキルの中心ワークフローは以下へ変更すべき。

1. 原稿取り込み
2. 章・話構造の正規化
3. キャラクター設定抽出
4. 章あらすじ生成
5. 画像生成定義の作成
6. キャラクター画像生成
7. 章扉絵生成
8. モバイル Web ビルド
9. 読み味と UI の確認、微修正

## 4. 新しい構成モデル案

### 正式な構成単位

今回の改修後は、本文と UI の正式単位を `章 > 話` にする。

- `章`: 扉絵と章あらすじを持つ大きな区切り
- `話`: 本文の表示・読書進行の単位

### 廃止または縮退するもの

- `節` を本文構成の必須単位にしない
- `scene-id` を前提にした節画像生成フローを外す
- 「節画像が Web 表示の最小単位」という前提をやめる

### UI 上の見せ方

- タイトルページ
- 章の開始時に扉絵
- 章タイトル
- 章あらすじ
- その章に属する話を順番に表示

## 5. 想定する成果物

改修後にスキルが生成・更新するべき主な成果物は次のとおり。

- `project/00_project_overview.md`
  - 作品情報、今回の変換対象、UI 方針
- `project/02_characters.md`
  - 原稿から抽出したキャラクター設定
- `project/04_chapter_outline.md`
  - `章 > 話` 構造の正規化結果
- `project/06_chapter_summaries.md`
  - 章ごとのあらすじ
- `project/manuscript/full_novel.md`
  - Markdown 整形済み本文
- `prompts/character-portraits.json`
  - キャラクター画像生成定義
- `prompts/chapter-cover-images.json`
  - 章扉絵生成定義
- `project/assets/characters/*`
  - キャラクター画像
- `project/assets/chapters/*`
  - 章扉絵
- `project/assets/ui/*`
  - 背景 SVG や UI 素材
- `docs/*`
  - モバイル Web 出力

## 6. スキル文書の変更方針

`soudoku-novel-builder/SKILL.md` は、以下の観点で全面改修すべき。

### First Moves の変更

- `草稿/初稿.md` のような完成原稿があるか確認する
- 原稿から章・話を抽出する
- 原稿の構造を正規化して `project/manuscript/full_novel.md` に変換する
- キャラクター設定と章あらすじを本文から抽出する
- 抽出結果をユーザ確認に出す

### Composition Model の変更

- `章 > 話 > 節` 固定を廃止
- 新しい正式単位を `章 > 話` にする
- `節` は必要時のみ補助概念として扱い、必須にしない

### Standard Loop の変更

- 初期設計中心ではなく、原稿解析中心へ変更
- 画像生成は「キャラクター画像」「章扉絵」を中心にする
- Web ビルドは章・話ベースで行う

### Commands の変更

- `build-episode-image-manifest.mjs`
- `generate-episode-image.mjs`

上記 2 つは中心コマンドから外す。代わりに次のようなコマンドを導入するのが自然。

- `node scripts/import-draft-manuscript.mjs 草稿/初稿.md`
- `node scripts/extract-characters-from-draft.mjs`
- `node scripts/build-chapter-summaries.mjs`
- `node scripts/generate-character-portrait.mjs <id|--all>`
- `node scripts/generate-chapter-cover.mjs <chapter-id|--all>`
- `node scripts/build-web-novel.mjs`

## 7. スクリプト変更案

### 7.1 追加したいスクリプト

#### `scripts/import-draft-manuscript.mjs`

役割:

- `草稿/初稿.md` を読み込む
- `第一章`, `第一話` を検出して正規化する
- `project/manuscript/full_novel.md` を `## 章`, `### 話` 形式で生成する
- `project/04_chapter_outline.md` の骨格も同時に生成または更新する

#### `scripts/extract-characters-from-draft.mjs`

役割:

- 本文から登場人物候補を抽出する
- 主要人物の外見、口調、役割、関係性、能力、反復描写をまとめる
- `project/02_characters.md` と `prompts/character-portraits.json` の下書きを作る

#### `scripts/build-chapter-summaries.mjs`

役割:

- 各章の全話をまとめて章あらすじを生成する
- `project/06_chapter_summaries.md` を生成する

#### `scripts/generate-chapter-cover.mjs`

役割:

- 章ごとの扉絵を生成する
- 出力先は `project/assets/chapters/`

### 7.2 変更したい既存スクリプト

#### `scripts/build-web-novel.mjs`

現状:

- 節画像ありきで `scene` を描画している
- `#### 節` と `scene-id` を前提にデータを組んでいる

変更後:

- `章 > 話` ベースの JSON を出す
- 話本文を読み物として流す
- 章の開始位置に扉絵を差し込む
- 本文背景用の SVG または装飾アセットを読み込む

#### `scripts/check-project-state.mjs`

現状:

- 節数、節画像、節参照などを確認している

変更後:

- 原稿取込済みか
- キャラクター抽出済みか
- 章あらすじ生成済みか
- 章扉絵定義と生成状況
- Web ビルド状況

を確認対象に変更する。

## 8. モバイル UI 改修案

### 現在の問題

- 現在の UI は節ごとに画像カードを並べる構成
- 今回の原稿には節がない
- 文章を読む体験より、場面ビジュアルの連打が前面に出ている

### 新 UI の方針

- 読書の主役を本文に戻す
- 画像は章の入口だけに使う
- 章メニューから章・話に移動できる
- 本文は軽い紙面感のある読みやすい背景で支える

### 表示構成案

1. タイトルページ
2. 章ヘッダー
3. 章扉絵
4. 章あらすじ
5. 章内の話一覧
6. 各話本文

### 背景デザイン案

- 画像ではなく SVG / CSS ベースの静かな背景を使う
- 候補は「和紙」「淡い木漏れ日」「薄いインクのにじみ」「柔らかい布地」
- 本文面は高コントラストだが硬すぎない配色にする
- 行間、字間、段落間隔はスマホ縦読みを優先する

### 章扉絵の扱い

- 各章の最初に 1 枚だけ表示する
- 話ごとには画像を挟まない
- 扉絵の上に章タイトルを重ねるかは、UI 実装時に見て判断する

## 9. 今回の改修で削るもの

今回の要件では、少なくとも以下は主役から外すべき。

- 節画像マニフェスト中心の運用
- 節単位の画像生成
- 「まずプロットを作ってから本文へ進む」というワークフロー
- ユーザ確認前に本文を書かない、という新規執筆前提のゲート

## 10. 実装順序案

承認後は、次の順で実装するのが安全。

1. `SKILL.md` と `references/project-files.md` を改修
2. 原稿取込スクリプトを作成
3. キャラクター抽出と章あらすじ生成スクリプトを作成
4. `build-web-novel.mjs` を章・話ベースへ改修
5. モバイル UI を章扉絵型へ改修
6. 背景 SVG / CSS を追加
7. `草稿/初稿.md` で一連の変換を実行して成果物を生成

## 11. 承認後に行う具体作業

承認をもらえたら、次を実施する。

- スキル文書の書き換え
- 参照文書の書き換え
- 原稿取込スクリプト追加
- 抽出系スクリプト追加
- Web ビルド改修
- モバイル UI 改修
- `草稿/初稿.md` を入力にした実行確認

## 12. 補足

この設計では、現行スキルの「創りながら読む」性格は薄くなる。一方で、「完成済み原稿を、画像付きで読めるモバイル小説として出版可能な形へ変換する」用途にはかなり適合する。

今回の依頼内容に最も忠実なのは、この方向だと判断している。
